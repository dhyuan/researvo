# Researvo

## MVP Requirements Document

Version: V0.0.1
Date: 2026-05-18
Status: Draft
Related PDD: `2026-05-18-V0.0.1-PDD.md`

---

## 1. MVP Goal

The MVP goal is to validate the core product thesis:

> Economics and social science students can use Researvo to create or import a research-grade survey schema, validate it, publish it by URL, collect anonymous responses, and export stable variable data for external analysis.

The MVP should be small in user-facing scope but architecturally aligned with the long-term PDD.

---

## 2. Target User

Primary target user:

> Economics and social science students conducting small research projects, coursework, thesis pilots, or independent surveys.

Primary respondent:

> A person who receives a survey URL and submits responses anonymously.

---

## 3. MVP Product Shape

The MVP is not a complete survey platform. It is a schema-first research survey workflow:

1. Research operator creates or imports a survey schema.
2. System validates and lints the schema.
3. Research operator previews the survey.
4. Research operator publishes a frozen survey version.
5. Respondents complete the survey through a URL.
6. System stores response data.
7. Research operator exports raw data and schema metadata.

---

## 4. In Scope

### 4.1 Survey Schema V0.0.1

The system must support a structured survey schema with these core sections:

- Survey metadata.
- Consent block.
- Question nodes.
- Variable definitions.
- Scale definitions.
- Coding schemes.
- Missing value rules.
- Branching logic.
- Randomization metadata.
- PII classification metadata.
- Survey policy.
- Version metadata.

The first schema version may be limited, but it should be explicit and versioned.

### 4.2 Schema Import

The research operator can import a survey schema as JSON.

Required behavior:

- Paste or upload JSON.
- Parse schema.
- Show parse errors.
- Show validation report.
- Store draft schema if valid enough to save.

XML can remain a future export/import format unless implementation cost is low.

### 4.3 Basic Manual Editing

The research operator can make basic edits without a full visual flow editor.

MVP editing can be form-based or JSON-assisted.

Required editing capabilities:

- Edit survey title and description.
- Edit consent text.
- Add, edit, delete question nodes.
- Edit variable names.
- Edit options and coding values.
- Edit required/optional behavior.
- Edit simple branch rules.

The full drag-and-drop visual flow editor is not required for MVP.

### 4.4 AI-Assisted Schema Generation

The research operator can use AI to generate a draft schema.

Required behavior:

- User enters research goal and survey constraints.
- User selects or configures an AI provider.
- AI generates survey schema JSON.
- System parses and validates AI output.
- AI-generated output must not be published without user review.

Supported provider architecture:

- OpenAI-compatible endpoint.
- Custom base URL.
- API key.
- Local endpoint such as Ollama.

Provider-specific presets for OpenAI, Claude, DeepSeek, and others can be added later.

### 4.5 Validator/Linter

The validator/linter is a core MVP feature.

The MVP must classify findings as:

- Error: blocks publish.
- Warning: allows publish but needs attention.
- Suggestion: optional improvement.

Required validation checks:

- Invalid JSON or schema structure.
- Missing required survey metadata.
- Missing consent block warning.
- Duplicate question IDs.
- Duplicate variable names.
- Missing variable names.
- Missing or invalid branch targets.
- Unreachable nodes.
- Dead-end paths without terminal state.
- Obvious cycles.
- Missing coding for closed-choice questions.
- Option labels without stable coding values.
- Missing scale metadata for scale questions.
- Missing missing-value policy warning.
- PII-like fields without PII classification warning.
- Randomization metadata without assignment logging warning.

AI-assisted lint checks may include:

- Leading question phrasing.
- Double-barreled questions.
- Sensitive question warning.
- Ambiguous answer options.

If AI linting is too costly for the first implementation, the architecture should still expose an interface for it.

### 4.6 Preview

The research operator can preview the survey before publication.

Required behavior:

- Render survey from schema.
- Traverse simple branch logic.
- Show consent page if present.
- Show validation status.
- Indicate that preview responses are not production data.

### 4.7 Publishing

The research operator can publish a survey version and receive a public URL.

Required behavior:

- Publishing freezes a survey version.
- Published version receives a stable URL or public ID.
- Draft schema can change after publication without changing the already published version.
- Publish is blocked if validator has blocking errors.

### 4.8 Respondent Runtime

Respondents can open a URL and complete the survey anonymously.

Required behavior:

- Render published survey version.
- Show consent block before questions if configured.
- Execute required validation.
- Execute simple branching.
- Record start time and completion time.
- Submit response.
- Show completion page.

### 4.9 Survey Policy

MVP default collection mode is anonymous.

Required architecture:

- Survey access must go through a policy/access gate.
- MVP policy can allow anonymous access by default.
- Policy model should reserve future support for OAuth, invite-only, organization verification, and proof-of-personhood.

MVP behavior:

- Anonymous access.
- Optional CAPTCHA can be deferred if implementation cost is high.
- Basic duplicate prevention can be cookie-based or omitted from the first internal prototype, but the policy object should exist.

### 4.10 Data Collection

The system stores response data tied to a published survey version.

Required data:

- Survey ID.
- Survey version ID.
- Respondent session ID.
- Response values.
- Missing value markers where applicable.
- Branch path or shown question IDs.
- Randomization assignments if randomization is used.
- Started at timestamp.
- Submitted at timestamp.
- Completion status.

### 4.11 Export

The research operator can export collected data.

Required MVP exports:

- CSV response export.
- JSON response export.
- JSON schema export.

CSV export should use stable variable names as column headers.

Export should preserve:

- Coding values.
- Missing value markers.
- Survey version ID.
- Submission timestamp.

### 4.12 Basic Metrics

The research operator can view basic survey collection metrics.

Required metrics:

- Number of started sessions.
- Number of completed submissions.
- Completion rate.
- Average completion time.

---

## 5. Out Of Scope For MVP

The MVP does not include:

- Full drag-and-drop visual flow editor.
- Enterprise permissions.
- Multi-user collaboration.
- Advanced dashboards.
- Built-in statistical modeling.
- AI-generated research reports.
- Real-time AI interview mode.
- Sample panel marketplace.
- Payment or billing.
- Organization accounts.
- Strong identity verification.
- World ID or other proof-of-personhood integration.
- Complex quota sampling.
- Complex workflow automation.
- Native Stata/SPSS/R package export.

These should remain possible through architecture, but should not be first-version requirements.

## 6. MVP Domain Model

The MVP should be designed around these domain objects:

- User.
- Survey.
- SurveyDraft.
- SurveyVersion.
- SurveySchema.
- QuestionNode.
- VariableDefinition.
- ScaleDefinition.
- CodingScheme.
- MissingValuePolicy.
- ConsentBlock.
- PIIClassification.
- SurveyPolicy.
- RespondentSession.
- SubmissionRecord.
- ValidationReport.
- ValidationFinding.
- AIProviderConfig.
- ExportJob or ExportProfile.

The implementation plan may simplify persistence, but these boundaries should guide code design.

---

## 7. Minimum Question Types

MVP should support a small but research-useful set of question types:

- Single choice.
- Multiple choice.
- Short text.
- Long text.
- Number.
- Likert scale.

Future question types:

- Ranking.
- Matrix.
- Date/time.
- File upload.
- Slider.
- Constant-sum.
- Experiment/treatment assignment block.

---

## 8. Minimum Branching Rules

MVP should support simple conditional branching:

- If answer equals value, go to node.
- Otherwise continue to next node.
- Terminal/end node.

Future branching:

- Multi-condition expressions.
- Random assignment.
- Quotas.
- Computed variables.
- Piped text.

---

## 9. Acceptance Criteria

The MVP is acceptable when the following workflows work end to end:

1. A research operator creates a new survey draft.
2. The operator imports or AI-generates a JSON survey schema.
3. The system validates the schema and shows errors, warnings, and suggestions.
4. The operator fixes blocking errors.
5. The operator previews the survey.
6. The operator publishes a frozen survey version.
7. A respondent opens the public URL and completes the survey anonymously.
8. The system stores the submission against the correct survey version.
9. The operator views basic collection metrics.
10. The operator exports responses as CSV and JSON.
11. The exported data uses stable variable names and coding values.
12. The original survey schema can be exported as JSON.

## 10. Open Questions

These questions should be resolved before or during implementation planning:

1. Should the first implementation require user accounts for research operators, or can local/single-user mode come first?
2. Should AI generation be mandatory in MVP, or can schema import plus provider interface come first?
3. What is the exact Survey Schema V0.0.1 JSON format?
4. How strict should publish-blocking validation be in the first version?
5. Should consent be required for all published surveys or only recommended?
6. Should anonymous duplicate prevention be implemented in the first build or left as policy metadata?
7. Which export format is the first priority after CSV and JSON?
8. Should template support be part of MVP or Phase 2?

## 12. Implementation Notes

Implemented decisions as of the first MVP build:

- Survey Schema V0.0.1 is defined in `lib/schema/surveySchema.ts` using Zod.
- Deterministic validation lives in `lib/validation/` and emits structured errors, warnings, and suggestions.
- Persistence uses Prisma 6 with PostgreSQL.
- Published versions are created through `lib/publishing/publishingService.ts` and are deep-cloned from drafts.
- Anonymous respondent access goes through `lib/runtime/accessGate.ts`.
- Runtime traversal uses `nextNodeId` and `branches[].goto`; visual `edges` are not executable traversal.
- AI generation uses an OpenAI-compatible adapter in `lib/ai/openaiCompatibleProvider.ts`.
- Exports include CSV response export, JSON response export, and schema JSON export.
- The E2E test is in `tests/e2e/mvp-flow.spec.ts` and requires a reachable `DATABASE_URL`.
