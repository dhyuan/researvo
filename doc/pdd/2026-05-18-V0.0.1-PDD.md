# Researvo

## 1. Product Summary

Researvo is a schema-first, AI-assisted, research-grade survey platform.

The product starts with a narrow first user group: economics and social science students who need to design, publish, collect, validate, and export structured survey data for research projects.

The long-term ambition is broader: to become human signal collection infrastructure for academic research, market research, UX research, public policy, product decision-making, and AI-era human data workflows.

The core product thesis:

> In the AI era, the scarce resource is not analysis software, but high-quality, structured, trustworthy human data.

Researvo is not primarily a dashboard, BI system, SPSS replacement, or AI report generator. It is the infrastructure layer that helps people create valid survey instruments and collect analysis-ready human data.

---

## 2. Why Now

Existing survey tools such as Google Forms, SurveyMonkey, Typeform, Qualtrics, and similar platforms are useful, but many still center on form creation, basic collection, and simple reporting.

AI changes the survey workflow in two directions:

1. Researchers can use AI to draft survey instruments faster.
2. AI-generated instruments can introduce new risks: inconsistent wording, unstable variables, hidden bias, invalid branching, and unreliable data.

Researvo addresses this gap by combining AI-assisted creation with strict research-grade structure:

- AI can help generate or revise survey schemas.
- The platform validates the schema before publication.
- Published survey versions keep variables, coding, and question definitions stable.
- Collected data can be exported into analysis-friendly formats.

---

## 3. Product Positioning

Researvo is:

> AI-compatible survey infrastructure for research-grade human data collection.

It is not:

- A general form builder only.
- A statistical analysis package.
- A BI platform.
- An AI auto-paper tool.
- A replacement for R, Python, Stata, SPSS, or Excel.

It complements existing analysis tools by producing cleaner, more structured, better documented raw data.

---

## 4. First Target User

The first target user is:

> Economics and social science students conducting course projects, thesis research, pilot studies, or small independent research projects.

This user group is valuable because they need research structure, but usually do not require enterprise-grade permissions, compliance, procurement, or workflow automation in the first version.

Their key needs:

- Design a survey around a research question.
- Define stable variables.
- Use scales, coding, and missing value rules correctly.
- Publish a survey by URL.
- Collect anonymous responses.
- Export raw data for analysis in R, Python, Stata, SPSS, Excel, or similar tools.
- Avoid common research design mistakes.
- Use AI to draft a survey while still retaining human review and methodological control.

## 6. Core Product Principles

### 6.1 AI Assists, But Does Not Define Statistical Standards

AI can:

- Draft survey schemas.
- Suggest questions.
- Suggest branching logic.
- Suggest wording improvements.
- Check for bias.
- Help repair invalid schemas.

AI should not silently:

- Change published core research questions.
- Change wording per respondent.
- Change answer options per respondent.
- Change variable definitions.
- Change coding schemes.

Formal research requires stable variables, comparable samples, repeatable instruments, and clear documentation.

### 6.2 Schema First

The core product object is not a web form. It is a survey schema.

The schema describes:

- Survey metadata.
- Consent blocks.
- Question nodes.
- Variable definitions.
- Scale definitions.
- Coding schemes.
- Missing value rules.
- Branching logic.
- Randomization rules.
- PII classifications.
- Survey policies.
- Version information.
- Export metadata.

The UI, runtime engine, validator, AI assistant, and export system all operate on this schema.

### 6.3 Data Collection Before Analysis

The first product phase prioritizes collecting structured, validated, analysis-ready data.

Advanced analysis, dashboards, AI-written reports, and automated insights are future layers.

### 6.4 Open Workflow

Researvo should not lock users into a proprietary analysis environment.

The platform should support exports that work well with:

- CSV.
- JSON.
- XML.
- R.
- Python.
- Stata.
- SPSS.
- Excel.

Future export profiles may include codebooks, variable dictionaries, and package-specific formats.

### 6.5 Trust Is A Survey Policy

Different surveys need different respondent trust levels.

Trust should be modeled as survey policy rather than hard-coded platform behavior.

MVP collection can be anonymous, while the architecture should allow future modes such as OAuth, invite-only access, organization verification, and proof-of-personhood.

---

## 7. Differentiation

Researvo's differentiation is:

1. AI-generatable open survey schema.
2. Research-grade schema validation and linting.
3. Stable variable model for research comparability.
4. Analysis-friendly export.
5. Extensible respondent trust policy.

The product should be described as an AI-compatible research data collection layer, not simply as an AI form builder.

## 8. Core Domain Concepts

### 8.1 Survey Schema

The structured definition of a survey. It is the source of truth for design, validation, publication, runtime behavior, and export.

### 8.2 Survey Version

A frozen version of a survey schema. Once published, a version should preserve variable definitions, question wording, coding, and logic so collected data remains comparable.

### 8.3 Question Node

A unit in the survey flow. A node may represent a question, consent block, section, instruction, randomization block, or terminal state.

### 8.4 Variable Definition

The analysis-facing representation of a question or derived value. It includes the variable name, label, type, coding, scale, missing value rules, and export behavior.

### 8.5 Scale

A measurement structure used to capture a concept. Examples include Likert scales, numeric ratings, rankings, semantic differential scales, and frequency scales.

### 8.6 Coding Scheme

The mapping between respondent-facing answers and analysis-facing values. For example, "Strongly disagree" may be coded as 1 and "Strongly agree" as 5.

### 8.7 Missing Value Policy

Rules that distinguish why a value is missing, such as not shown due to branch logic, skipped by respondent, refused, unknown, timeout, or system error.

### 8.8 Randomization

Rules that randomize question order, option order, or treatment assignment. Randomization must be recorded so analysis remains reproducible.

### 8.9 Consent

The respondent-facing informed consent content that explains the study purpose, data use, anonymity, withdrawal rights, and contact information.

### 8.10 PII Classification

A classification of fields that may identify a person, such as name, email, phone number, student ID, address, IP address, or similar attributes.

### 8.11 Survey Policy

The access, trust, privacy, duplicate submission, CAPTCHA, rate limit, and data handling rules for a survey.

### 8.12 Validation Report

A structured report generated by the validator/linter, showing errors, warnings, suggestions, and research design concerns.

## 9. Schema Extensibility

Different fields and disciplines need different metadata. Researvo should provide a standard core metadata model while allowing controlled extension.

The schema should support:

- Required core fields for all surveys.
- Optional research metadata fields.
- Domain-specific templates.
- User-defined custom properties.
- Validation rules for both core and custom fields.

This allows the platform to preserve a research-grade standard while avoiding a rigid one-size-fits-all schema.

---

## 10. AI Strategy

Researvo should support built-in AI assistance, but AI output must pass through schema parsing, validation, and human review before publication.

The AI flow:

1. User provides a research goal, topic, constraints, or draft questionnaire.
2. AI generates or edits a survey schema.
3. Platform parses the schema.
4. Validator/linter checks the schema.
5. User reviews and fixes issues.
6. User publishes a frozen survey version.

The provider model should be pluggable:

- OpenAI-compatible endpoint.
- OpenAI.
- Claude.
- DeepSeek.
- Ollama or other local endpoint.
- Custom base URL and API key.

The product should be AI-neutral and should not depend on a single model provider.

---

## 11. Validator And Linter

The validator/linter is a core product feature.

It should check at least:

- Dead ends.
- Cycles.
- Unreachable nodes.
- Unreferenced nodes.
- Invalid branch targets.
- Variable name conflicts.
- Option coding changes.
- Missing coding schemes.
- Missing scale metadata.
- Incomplete missing value rules.
- Branch coverage issues.
- PII fields without classification.
- Consent requirements.
- Sensitive question warnings.
- Leading or biased phrasing.
- Double-barreled questions.
- Randomization without assignment logging.
- Version compatibility issues.

Validation results should be structured so the UI, AI assistant, and export pipeline can all consume them.

---

## 12. Trust System

Trust should be designed as a pluggable policy layer.

Future trust levels may include:

| Level | Description |
|---|---|
| Anonymous | No respondent login required |
| Soft Identity | Email or OAuth identity |
| Verified Human | CAPTCHA plus risk checks |
| Strong Identity | OAuth plus phone or stronger verification |
| Organization Verified | School or company account |
| Proof of Personhood | World ID or similar proof-of-personhood provider |

MVP should default to anonymous collection while routing survey access through an access gate that can support future policies.

- Topic extraction.
- Sentiment analysis.
- Clustering.
- Basic reporting.

- Experiment management.
- Advanced consent workflows.
- Organization accounts.
- Panel/sample integrations.
- Full research lifecycle management.
